Presentation

Deep Research Agent

Graph-based research workflow with LangGraph

Section-oriented architecture: plan → parallel section workers → merge → conflict handling → writer → report

Default: Phase 2 (section workers). Phase 1 (single-agent loop) kept for comparison and debugging.

Overview

What It Does — One-Liner

A graph-based deep research agent that takes a question, plans a report, runs parallel section workers (each with its own search → normalize → coverage loop), merges evidence, optionally resolves conflicts, curates context for the writer, and produces a grounded markdown report with citations.

Capabilities

  • Accepts a research question from the CLI (or stdin)
  • Classifies query complexity and creates a research plan
  • Optionally pauses for plan approval before expensive research
  • Runs web search across Exa, Tavily, or Gensee
  • Normalizes and deduplicates evidence
  • Prepares a capped writer context (no dump-everything)
  • Writes report to reports/; optional trace JSON, logs, evals
Architecture

Default Flow (Phase 2) — High-Level

ingest_request → classify_complexity → create_research_plan → [Human-in-the-loop: approve / edit / cancel] → decompose_into_sections → dispatch_sections → section_worker × N (parallel) → merge_section_evidence → detect_global_gaps_and_conflicts → conflict_route → [optional] conflict_resolution_research → prepare_writer_context → write_sections → write_report → finalize_messages

Why a graph (LangGraph)?

We need loops (research until coverage or budget), branching (conflict vs no conflict), and a clear interrupt (plan approval). A graph makes control flow explicit and debuggable.

State

State and Reducers (Why They Matter)

Parallel section workers each return state updates. Reducers define how those updates merge.

FieldReducerReason
section_resultsoperator.addAccumulate one result per worker
global_seen_urls_merge_setsUnion URLs across workers
query, section_max_iterations_keep_firstParallel workers must not overwrite shared scalars
messagesadd_messagesAppend conversation

Writer fallback

If writer_evidence_subset is empty (e.g. reducer/merge order), the writer node uses merged_evidence so the report always has sources. This fallback exists because we hit the “writer got no evidence” case in practice.

Design

Design Decisions You Can Defend

  • Graph instead of chain: Routing, loops, shared state, optional interrupt — explicit and inspectable.
  • Normalize evidence: Search is noisy; normalization improves report quality more than many prompt tweaks.
  • Cap writer context: Too much context hurts quality; ranking and capping beats “dump everything.”
  • Section workers: Reports are sectional; per-section research = narrower queries and better coverage for broad topics.
  • Pause after plan only: Human leverage where it matters (scope) without making the whole flow interactive.
  • Explicit coverage + iteration budget: “Search until done” is not a design; coverage checks make the loop predictable.
Honesty

Known Shortcomings (From DESIGN)

  • Config and configuration.py not fully aligned for some planner model names.
  • Section summaries don’t get full evidence body yet.
  • Conflict resolution is still shallow.
  • Dedup is URL-based (no semantic dedup yet).
  • Tests are more graph/smoke than deep node correctness.

What would you improve next?

Config cleanup, stronger section summaries (full evidence into summary), better conflict adjudication, semantic dedup, caching for search, streaming report output, more unit tests per node.

Interview

Interview Q&A — Highlights

“How do you avoid the writer hallucinating or ignoring sources?”

We normalize and curate evidence, map it to sections, and cap what the writer sees. We repair the Sources section from state when needed. Evals include claim support and citation relevance.

“How do you handle conflicting information?”

We detect global conflicts after merging section evidence. If enabled, we run conflict_resolution_research before preparing writer context; the writer then sees the reconciled picture.

“What was the hardest bug?”

State merging with parallel section workers. Reducers (operator.add, _merge_sets, _keep_first) and checkpointer merge order could leave writer_evidence_subset empty — we added a writer fallback to merged_evidence.

Evals & Config

Evaluation & Configuration

Evals (with --eval)

Claim support, factual accuracy, citation relevance, source quality, section completeness, comparative breadth, synthesis quality, conflict handling, stop decision quality, tool trajectory quality. Used for inspection and benchmarking, not hard gates.

Config (config.yaml)

Search provider/depth, max iterations, writer context size, full-page enrichment, section-worker settings, conflict-resolution behavior, model choices, prompt overrides, report structure. CLI overrides config.
